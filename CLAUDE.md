## プロジェクト概要

不動産アプリ (realestate-app)。物件情報の検索・管理・表示を行うWebアプリケーション。

---

## Git 運用ルール

**コードを変更するたびに、必ずGitHubへプッシュすること。**

具体的な手順:

1. 変更内容をステージングする
   ```
   git add <変更したファイル>
   ```
2. 意味のあるコミットメッセージでコミットする
   ```
   git commit -m "feat: 変更内容の簡潔な説明"
   ```
3. リモートへプッシュする
   ```
   git push origin <現在のブランチ>
   ```

### コミットメッセージの規約

[Conventional Commits](https://www.conventionalcommits.org/) に従う:

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング（動作変更なし） |
| `style:` | コードスタイルの修正 |
| `docs:` | ドキュメントの変更 |
| `test:` | テストの追加・修正 |
| `chore:` | ビルドやツール設定の変更 |

### ブランチ戦略

- `main` — 本番リリース用。直接プッシュ禁止。
- `develop` — 開発統合ブランチ。
- `feat/<機能名>` — 機能開発ブランチ。
- `fix/<バグ名>` — バグ修正ブランチ。

### 禁止事項

- `git push --force` を `main` ブランチへ実行しない。
- `--no-verify` でフックをスキップしない。
- 機密情報（APIキー、パスワード）をコミットしない。

---

## コーディング規約

- 変更を依頼されていない箇所は修正しない。
- 不要なコメント・docstring・型アノテーションを追加しない。
- 存在しないシナリオ用のエラーハンドリングを追加しない。
- 新しいファイルは必要な場合のみ作成する（既存ファイルの編集を優先）。
- セキュリティ脆弱性（SQLインジェクション、XSS等）を絶対に導入しない。

---

## 作業フロー

1. コードを読んで理解してから変更を提案する。
2. 変更はスコープを最小限に保つ。
3. **変更後は必ずGitHubへプッシュする。**
4. UIの変更はブラウザで動作確認してから完了を報告する。
